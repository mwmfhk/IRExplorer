Section 01 — DSP

Polygon IR Blending via Inverse-Distance Weighting

Each loaded IR sits at a vertex of a regular polygon. The blend point — the orange dot you drag — sits somewhere inside. Each IR's contribution to the mix is determined by how close the blend point is to its vertex: closer means louder.

The classic formulation is inverse-distance weighting (Shepard, 1968). For N vertices vi and a blend point p:

di = ‖p − vi‖
wi = (1/di) / Σj (1/dj)

This gives non-negative weights that sum to 1. At a vertex, di → 0, so wi → 1 and every other weight collapses to zero. Move the blend point along the centroid and you get an even split.

function slotWeights(slots, bp) {
  const verts = slotVertices(slots);
  const eps   = 1e-6;  // avoid divide-by-zero exactly at a vertex
  const invD  = verts.map(v => 1 / (Math.hypot(bp.x - v.x, bp.y - v.y) + eps));
  const sum   = invD.reduce((a, b) => a + b, 0);
  return invD.map(w => w / sum);
}
Blend point dropped near vertex 1 produces weights heavily favoring IR 1.

The + eps guard prevents NaN when you drop the blend point exactly on a vertex. Without it, one term goes infinite and the normalization explodes.

Section 02 — DSP

Phase Correction

Two cabinet IRs captured with different mics (or even the same mic at different positions) have their main impulses arriving at different sample indices. Sum them naively and you get comb filtering: a frequency response littered with notches.

What comb filtering looks like

Suppose IR A peaks at sample 0 and IR B peaks at sample Δt. Their weighted sum has transfer function:

H(f) = a + b · e−i 2πf Δt
|H(f)|² = a² + b² + 2ab · cos(2π f Δt)

That cosine puts a notch every 1/(2Δt) Hz and a peak every 1/Δt Hz. For Δt = 10 samples at 48 kHz, that's a notch every 2.4 kHz — exactly in the band where guitar tone lives.

Two delta-impulse IRs offset by Δt produce a comb-filter magnitude response.

Mode 1: No Correction

Raw weighted sum. Default. Preserves each IR's original phase character — many guitarists prefer this for the "air" and "smear" of each cab's natural response, accepting the comb filtering as an artifact of blending.

Mode 2: Align Peak

Find the index of each IR's absolute maximum sample and shift each one so all peaks land at the same index (the latest one across all IRs). Cheapest fix that handles the most common case (different mic distances).

const peaks  = slots.map(s => peakIdx(s.buffer.getChannelData(0)));
const target = Math.max(...peaks);
// pad slot i with (target - peaks[i]) zeros at the front

Mode 3: Cross-Correlation Alignment

Peak alignment can miss when the most-prominent sample isn't actually representative of the IR's "main arrival". Cross-correlation looks at the full waveform similarity:

r[k] = Σn ref[n] · y[n+k]

The lag k* where r peaks tells you how far y trails the reference. Shift y left by k* samples to align. Implemented via FFT for O(N log N):

r = IFFT( FFT(ref) · conj(FFT(y)) )
Cross-correlation locates the lag at which the two waveforms best line up.

Mode 4: Minimum-Phase

The other modes shift IRs in time. Minimum-phase goes deeper: it rebuilds each IR so its magnitude response is identical but its phase is reconstructed to concentrate all energy as causally early as possible. After conversion, every IR has its main energy at sample ~0, so blending them adds without comb filtering — by construction.

The recipe via the real cepstrum:

  1. X = FFT(x)
  2. Take log|X| (real-valued log magnitude)
  3. c = IFFT(log|X|) — this is the real cepstrum
  4. Fold: keep c[0] and c[N/2], double c[1..N/2−1], zero the anti-causal half
  5. FFT the folded sequence → complex log-spectrum with minimum-phase imaginary part
  6. exp(complex) → minimum-phase spectrum
  7. IFFT → minimum-phase time-domain IR
Min-phase conversion preserves the magnitude response exactly while concentrating energy at sample 0.
Trade-off: minimum-phase eliminates inter-IR comb filtering but erases each IR's original phase character — the subtle decay shape that some hear as "naturalness". The default mode is No Correction for this reason. Use min-phase when you want a clean, predictable blend; use raw when you want every IR's full personality.

Section 03 — DSP

Cooley-Tukey Radix-2 FFT

Both cross-correlation and min-phase rely on FFTs. We implemented a radix-2 Cooley-Tukey FFT in ~30 lines of pure JavaScript — no external library — that runs in-place on parallel Re/Im Float32Arrays.

The decimation-in-time algorithm splits an N-point DFT into two N/2-point DFTs (even-indexed and odd-indexed samples), combined by N/2 butterflies:

y[k] = x[k] + WNk · x[k + N/2]
y[k + N/2] = x[k] − WNk · x[k + N/2]
where WN = e−i 2π/N

Applied recursively this gives log2(N) stages, each consisting of N/2 butterflies — total O(N log N).

To run in-place, the input has to be bit-reverse permuted first. For N = 8, that swaps indices [0,1,2,3,4,5,6,7] → [0,4,2,6,1,5,3,7]. Then three stages of butterflies follow:

Bit-reversed input flows through log₂(N) butterfly stages.

The trick of computing twiddle factors WNk incrementally inside the inner loop avoids per-step sin/cos calls — each butterfly just rotates the running twiddle by one step:

const ang = (inverse ? 2 : -2) * Math.PI / len;
const wRe = Math.cos(ang), wIm = Math.sin(ang);
for (let i = 0; i < n; i += len) {
  let cRe = 1, cIm = 0;            // current twiddle, starts at W^0
  for (let k = 0; k < half; k++) {
    // ... butterfly using cRe, cIm ...
    const nRe = cRe * wRe - cIm * wIm;   // incremental rotation
    cIm = cRe * wIm + cIm * wRe;
    cRe = nRe;
  }
}

Section 04 — DSP

K-Weighted Loudness Normalization

When you load multiple IRs, they're captured at all sorts of levels. The tool auto-computes a per-IR gain (normGainDb) so each one contributes at comparable perceived loudness, not just sample-peak.

The measurement standard is ITU-R BS.1770 K-weighting — the same loudness model used by broadcast TV and streaming platforms (LUFS / LKFS). It applies two biquad filters in series before measuring RMS:

  1. Pre-filter — a high-shelf at ~1.7 kHz boosting +4 dB, modeling the head/torso resonance of an average listener
  2. RLB filter — a high-pass at 38 Hz, removing inaudible low-frequency rumble from the measurement
Two biquad filters in cascade approximate human loudness perception; the integrated signal energy gives LUFS.

Final loudness:

LUFS = −0.691 + 10 · log10(mean(y²))

The IR's normGainDb is then the offset that brings its measured LUFS to a target reference, so all IRs contribute equally on average. The peak-normalization step that follows export is independent — it just prevents clipping in the final WAV.

Section 05 — Audio sync

Rhythm + Lead Phase-Locked Looping

The most subtle piece of the whole app. Here's the scenario:

  1. You record a rhythm part on the left panel and press Play. It loops over a region [loopStart, loopEnd] of period TL.
  2. While that loop plays, you press Record on the right panel and capture a lead. Recording starts mid-loop — some seconds φ into the rhythm's current cycle.
  3. When recording stops, the lead is, in general, not an integer multiple of TL long.
  4. Both loops must now play in phase forever — every time the lead loops back to its start, the rhythm must be at exactly the same phase φ it was when recording began.

The fix: pad the lead recording's tail with silence so its loop period becomes the smallest integer multiple of TL that covers it.

TL = (loopEnd − loopStart) × leftBuffer.duration
φ = (rightRecordStartCtxTime − playStart) mod TL
DR = rightAudioBuffer.duration (raw recording length)
N = ⌈ DR / TL ⌉
paddedDuration = N × TL

Build a new AudioBuffer of length paddedDuration, copy the raw recording into the first DR seconds, leave the rest as zeros. Set loop = true, loopEnd = paddedDuration. Every time the lead's transport hits paddedDuration, it wraps to 0 — and at that exact instant the rhythm has completed N full cycles since φ, so it's back at phase φ. Sync preserved indefinitely.

Padding the lead to N×T_L guarantees phase-lock with the rhythm forever.

Two start scenarios:

Auto-start (right just finished, left still playing):

elapsed     = audioCtx.currentTime - rightRecordStartCtxTime;
startOffset = elapsed % paddedDuration;
rightSource.start(0, startOffset);   // join mid-buffer at the right phase

Manual play (both already recorded, user hits Play):

playStart = ctx.currentTime;
leftSource.start(0, loopStart * leftBuffer.duration);
rightSource.start(playStart + φ, 0);   // wait φ seconds into the rhythm loop

Section 06 — Audio sync

Live Convolver Buffer Hot-Swap

The Phase Correction dropdown updates the playback without rebuilding the audio graph. The key Web Audio API trick: ConvolverNode.buffer is writable at any time. Assigning a new AudioBuffer to it instantly swaps the IR being convolved with the live input.

The state machine:

  • Each slot carries a cached slot.alignedBuffer — the AudioBuffer produced by applying the current alignment mode
  • A helper effectiveIRBuffer(slot) returns slot.alignedBuffer || slot.buffer
  • Every convolver-creation site uses conv.buffer = effectiveIRBuffer(slot)
  • syncConvolvers(side) recomputes slot.alignedBuffer for every slot, then walks the live convolver array and reassigns conv.buffer

syncConvolvers is called after every slot mutation (load, replace, remove), at the top of startPlay, and on the dropdown's change event:

function syncConvolvers(side) {
  const slots = side === 'left' ? irSlots : rightIrSlots;
  const mode  = side === 'left' ? leftAlignMode : rightAlignMode;
  if (slots.length > 0) {
    const ctx = getAudioCtx();
    const aligned = applyAlignment(slots, mode);  // returns [{channels:[…]}, …]
    slots.forEach((slot, i) => {
      const ch = aligned[i].channels;
      const buf = ctx.createBuffer(ch.length, ch[0].length, ctx.sampleRate);
      for (let c = 0; c < ch.length; c++) buf.getChannelData(c).set(ch[c]);
      slot.alignedBuffer = buf;
    });
  }
  if (isPlaying) {
    const convs = side === 'left' ? slotConvs : rightSlotConvs;
    convs.forEach((conv, i) => {
      if (conv && slots[i]?.alignedBuffer) conv.buffer = slots[i].alignedBuffer;
    });
  }
}
Changing the dropdown swaps the convolver's buffer in place; the graph stays connected.

The swap can be heard as a tiny click in the output (the impulse response changes instantaneously), but that's unavoidable without an asynchronous worker or a crossfade — which would defeat the point of an instant A/B.

Section 07 — Audio sync

Latency Calibration (Play-Along)

When the user plays a lead along to the rhythm loop, several offsets stack up between "the rhythm sample the user hears" and "the sample landing in the DI capture buffer":

  • Output latency — time from when a sample is scheduled to when the speaker actually emits it
  • Human reaction — how late (or early, for trained players) the user plays relative to the click they hear
  • Input latency — time from the DI signal hitting the audio interface to the sample arriving in the AudioWorklet

Total round-trip is wildly device- and player-dependent. If you don't compensate, the recorded lead plays back late versus the rhythm by exactly this amount.

Why not measure with a mic? The natural impulse here would be to play a click out the speaker and pick it up acoustically with a microphone — but the user is recording direct input (piezo or magnetic pickup straight into the interface). The DI line is acoustically dead — it sees only the strings, not the room. So the calibration has to involve the human.

The procedure

The system plays a metronome of sharp clicks (4 count-in beats so the user can lock onto the tempo, then several measurement beats). The user plays a note on the string for each click — exactly when they hear it.

For each measurement click at time tclick, the system inspects a window of the captured DI signal around it, finds the loudest sample (the user's note), and records the offset:

Δi = (peakIndexi / sampleRate + recordStart) − tclick,i

After N measurements, the system rejects outliers more than 1.5σ from the mean (sloppy beats, missed strums, room thumps), then averages the rest:

τ = mean( { Δi : |Δi − mean₀| ≤ 1.5 · σ₀ } )
User plays a DI note on each click; offsets between scheduled clicks and captured peaks are averaged.

The measured τ is stored as rightSyncTrimSec and applied when starting the lead during playback — the lead's effective start time is shifted earlier by τ seconds so what the user played in time with the rhythm now plays back in time with the rhythm.

What τ actually measures. A perfectly tight player produces τ ≈ output_latency + input_latency (pure audio round-trip). A player who reacts after hearing each click adds their reaction time on top — that's still the correct compensation, because they'll also play the actual lead the same way (reacting to what they hear). The calibration matches the human-in-the-loop offset, not just the audio chain.

Why a click impulse

A click is wideband, has a sharply localized peak in time, and is easy to play along to. It's also easy to detect — just find argmax|x[n]| in the per-beat window. A chirp would give better noise immunity but require deconvolution to recover the impulse time. For this human-paced measurement, the click is well above noise floor and the sample-level peak is plenty precise.

Section 08 — Audio sync

AudioWorklet Recorder vs MediaRecorder

The browser offers two ways to capture mic input:

  • MediaRecorder — produces encoded blobs (typically Opus in WebM container). Decoded asynchronously on the main thread. Convenient but lossy, with ~50–100 ms of buffering and no sample-accurate timing relative to the audio context clock.
  • AudioWorklet — a custom JS processor runs on the audio thread. It receives input as raw Float32Array blocks at the context's native sample rate, with sample-accurate timestamps via currentFrame.

For latency calibration (which depends on sample-accurate timing) and for clean lossless captures, AudioWorklet wins. The tool uses a tiny custom processor that just buffers input frames and posts them to the main thread via the worklet's port:

class RecorderProcessor extends AudioWorkletProcessor {
  process(inputs) {
    const ch = inputs[0]?.[0];
    if (ch) this.port.postMessage(new Float32Array(ch));  // copy to main
    return true;  // keep running
  }
}
registerProcessor('recorder-processor', RecorderProcessor);
RecorderProcessor copies raw samples on the audio thread; main thread accumulates them.

The main thread accumulates the Float32Array chunks into one big array and converts to an AudioBuffer when recording stops. No encode/decode round-trip, no Opus quantization, no surprise latency.

Section 09 — iOS

Web Audio Session Category Unlock

On iOS Safari, a freshly created AudioContext is silently assigned to the "ambient" audio session category. ctx.resume() succeeds, ctx.state returns 'running', ctx.currentTime advances, and the media volume indicator appears in the status bar — but no sound comes out of the speaker.

The standard unlock tricks (resume in a user gesture; play a 1-sample silent buffer) address whether the context is running, not which session category it's in. The session category is independent.

The fix: play an HTML5 <audio> element first. HTML5 audio elements are bound to the "media playback" category — and once that category is active for the page, the AudioContext inherits it.

Build a short silent WAV as a data URL, wrap it in an Audio element, play it (looped) in the first user gesture, and keep the element alive for the page's lifetime so the elevated session isn't released:

let iosSilenceAudio = null;
function unlockAndResume() {
  if (!iosSilenceAudio) {
    iosSilenceAudio = new Audio(buildSilenceWavDataUrl());
    iosSilenceAudio.loop = true;
  }
  iosSilenceAudio.play().catch(() => {});  // elevates session category
  const ctx = getAudioCtx();
  ctx.resume().catch(() => {});
  // ... standard 1-sample unlock buffer, just to be safe ...
}
iOS routes Web Audio to silent unless an HTML5 audio element has elevated the page's session category.

Section 10 — iOS

Sample-Rate Mismatch Handling

A session saved on a Mac (44.1 kHz AudioContext) loaded on an iPhone (48 kHz AudioContext) hits a strict sample-rate check:

Buffer sample rate does not match the context's sample rate

iOS Safari enforces sample-rate matching on ConvolverNode.buffer and AudioBufferSourceNode.buffer — and OfflineAudioContext enforces the same check, so it can't be used to resample around the problem.

The fix: pure-JS linear interpolation resampling. Read the raw Float32Array samples, resample to the destination context's rate, then create the AudioBuffer directly at that rate. No Web Audio API involved in the resample step, so the sample-rate check never fires.

function jsResample(input, inRate, outRate) {
  if (inRate === outRate) return input;
  const ratio = inRate / outRate;
  const n = Math.round(input.length / ratio);
  const out = new Float32Array(n);
  for (let i = 0; i < n; i++) {
    const s  = i * ratio;
    const lo = s | 0;
    const hi = Math.min(lo + 1, input.length - 1);
    out[i] = input[lo] + (input[hi] - input[lo]) * (s - lo);
  }
  return out;
}
Each output sample is a fractional position in the input array; linear interpolation between bracketing samples.

Linear interpolation is crude compared to a windowed-sinc resampler. It rolls off some high frequencies and adds tiny aliasing. But for IRs of guitar cabinets — where the relevant content is below 5 kHz and the IR's own response masks the artifacts — it's perceptually transparent. And it always works, on every browser, with no sample-rate surprises.

Section 11 — iOS

Pointer Events for Unified Mouse/Touch Dragging

Every drag widget — gain knobs, EQ knobs, the blend point, the waveform loop handles, the calibration drag line — needs to work with both mouse and touch. iOS touchscreens don't fire mousedown / mousemove / mouseup during a drag, so a mouse-only handler is dead on iPhone. Parallel touchstart / touchmove / touchend handlers would work, but each event's e.touches[0] shape differs from MouseEvent and every interaction would need to be wired twice.

The Pointer Events API unifies mouse, touch, and stylus into a single normalized event stream:

Pointer Events unify both input modalities under one API.

The handler API is identical to mouse events — 'pointerdown' / 'pointermove' / 'pointerup' in place of the 'mouse*' equivalents, with the same clientX / clientY properties on the event object.

iOS interprets touch drags on a normal element as a request to scroll or pinch-zoom the page. To suppress that and let JS receive the gesture, every draggable element needs touch-action: none in CSS:

.ir-knob, .input-knob, .norm-knob-canvas,
.blend-canvas, .cal-result-canvas, .region-handle {
  touch-action: none;
}

Combined with pointer-event handlers, all drag widgets work identically on Mac, Windows, iPhone, and iPad with no code branching.
